# Proj4025: Configure static code analysis in a .globalconfig file
No global analyzer configuration file (`.globalconfig`) was resolved during the
project evaluation phase. 

Using a global configuration via `.globalconfig` allows to centrally define
which rules are part of the [definition of done](https://en.wikipedia.org/wiki/Scrum_%28project_management%29#Definition_of_Done_%28DoD%29)
(or whatever code quality gate the team agreed on). Furthermore, by separating
this configuration from the `.editconfig` is considerd a good seperation of
concerns.

Using a global configuration via `.globalconfig` allows a team to centrally
define which static analysis rules are part of their [Definition of Done](https://en.wikipedia.org/wiki/Scrum_%28project_management%29#Definition_of_Done_%28DoD%29)
(or whichever quality gate the team has agreed upon). 

Furthermore, separating this compiler-level diagnostic configuration from the
IDE-focused `.editorconfig` file is considered a good [separation of concerns](https://en.wikipedia.org/wiki/Separation_of_concerns).
While `.editorconfig` excels at formatting rules (tabs, spaces, encoding, line-endings),
`.globalconfig` is built purely for compiler and analyzer behavior.

There are also edge cases where certain diagnostic severities defined in an
`.editorconfig` (such as [SYSLIB1045](https://learn.microsoft.com/dotnet/fundamentals/syslib-diagnostics/syslib1040-1049))
are silently ignored by the compiler, but are correctly applied when defined
inside a `.globalconfig`.

## Compliant
Place a file named `.globalconfig` in the root directory of your repository.
Ensure it contains the following entry at the top:
```ini
is_global = true
```
or, if your global configuration file uses a custom name or resides in a
specific folder, explicitly add it to your project or `Directory.Build.props`
file using the `GlobalAnalyzerConfigFiles` item group:

```xml
<ItemGroup>
  <GlobalAnalyzerConfigFiles Include="{some path}" />
</ItemGroup>
```
